## 1. Введение / Обзор

**Видеочат-комната** — это веб-приложение для группового видеозвонка со встроенным текстовым чатом, рассчитанное на **до 4 участников** в одной комнате одновременно. Пользователь открывает приложение, вводит отображаемое имя, создаёт комнату и делится ссылкой-приглашением; остальные присоединяются по этой ссылке. Внутри комнаты все участники видят и слышат друг друга в реальном времени и могут переписываться в общем чате.

**Проблема, которую решаем.** Небольшим группам (команде, друзьям, кандидату и интервьюеру) нужен мгновенный способ созвониться «лицом к лицу» прямо в браузере — без регистрации, установки приложений и сложной настройки. Существующие «тяжёлые» сервисы требуют аккаунтов и избыточны для быстрого звонка вчетвером.

**Решение.** Лёгкое приложение «зашёл по ссылке — представился — общаешься»: видео + аудио через WebRTC, текстовый чат и системные события через Socket.io, без авторизации и без серверного хранения истории.

---

## 2. Цели

1. Обеспечить групповой аудио-видеозвонок **до 4 участников** в одной комнате в реальном времени.
2. Дать любому пользователю создать комнату и пригласить других **одной ссылкой**, без регистрации.
3. Предоставить **общий текстовый чат**, синхронный с видеозвонком, видимый всем участникам комнаты.
4. Дать участнику контроль над приватностью: **включение/выключение микрофона и камеры** с понятной индикацией для других.
5. Обеспечить **осведомлённость о сессии**: кто в комнате сейчас, кто присоединился и кто вышел.
6. Корректно обрабатывать сбои и крайние случаи: отказ/отсутствие медиаустройств, обрыв соединения, недоступность сервера, неподдерживаемый браузер, попытку входа 5-го участника.
7. Работать в современных десктоп-браузерах (Chrome / Firefox / Edge 100+) на экранах шириной от 1024px.

## 3. User Stories

> Каждая история сопровождается секцией **Acceptance Criteria** в формате Gherkin (Given / When / Then), включая ошибочные и крайние сценарии.

### US-1 — Ввод и валидация имени
**As a** участник, **I want to** указать отображаемое имя до входа в комнату, **so that** другие участники узнавали меня в звонке и чате.

```gherkin
Scenario: Имя введено корректно
  Given пользователь открыл приложение
  When он вводит имя и нажимает "Создать комнату" или "Войти"
  Then он попадает в комнату
  And его имя отображается другим участникам на видео и в чате

Scenario: Пустое имя
  Given поле "Имя" пустое или состоит только из пробелов
  When пользователь пытается войти
  Then вход не происходит
  And показывается подсказка о необходимости ввести имя

Scenario: Слишком длинное имя или спецсимволы
  Given пользователь вводит имя длиннее 30 символов или со спецсимволами
  Then имя не принимается в таком виде (обрезается до 30 символов, спецсимволы запрещены/экранируются)

Scenario: Одинаковые имена в одной комнате
  Given в комнате уже есть участник с именем "Алекс"
  When входит другой участник и тоже указывает имя "Алекс"
  Then оба находятся в комнате под именем "Алекс"
  And система различает их по уникальному внутреннему идентификатору, который не показывается в интерфейсе
```

### US-2 — Создание комнаты
**As a** инициатор, **I want to** создать новую комнату, **so that** я мог начать звонок и пригласить в него людей.

```gherkin
Scenario: Успешное создание комнаты
  Given пользователь ввёл имя на стартовом экране
  When он нажимает "Создать комнату"
  Then система генерирует уникальный идентификатор комнаты
  And перенаправляет пользователя на URL с этим идентификатором
  And пользователь становится первым участником комнаты
```

### US-3 — Приглашение по ссылке
**As a** инициатор, **I want to** скопировать ссылку-приглашение, **so that** я мог отправить её другим людям любым способом.

```gherkin
Scenario: Копирование ссылки-приглашения
  Given пользователь находится в созданной комнате
  When он нажимает кнопку копирования ссылки
  Then URL текущей комнаты копируется в буфер обмена
  And пользователь видит подтверждение, что ссылка скопирована
```

### US-4 — Вход по ссылке
**As a** приглашённый участник, **I want to** открыть ссылку-приглашение и войти в комнату, **so that** я мог присоединиться к звонку.

```gherkin
Scenario: Вход в существующую комнату
  Given пользователь открыл URL с идентификатором активной комнаты
  When он вводит своё имя и подтверждает вход
  Then система считывает идентификатор из URL и подключает его к этой комнате
  And он видит и слышит остальных участников

Scenario: Вход по ссылке на несуществующую (удалённую) комнату
  Given комната с этим идентификатором не существует — не создавалась или была удалена
  When пользователь открывает URL с этим идентификатором и вводит имя
  Then система моментально создаёт новую комнату с тем же идентификатором
  And пользователь входит в неё как первый участник
  # Отдельного состояния "комната не найдена" не существует — любой URL создаёт или открывает комнату

Scenario: Вход по угаданному идентификатору
  Given существует активная комната
  When посторонний пользователь вводит/угадывает её идентификатор и открывает URL
  Then он попадает в этот звонок — это штатное поведение (доступ к комнате не ограничивается)
```

### US-5 — Ограничение в 4 участника
**As a** участник, **I want to** быть уверенным, что в комнате не больше 4 человек, **so that** качество звонка оставалось стабильным.

```gherkin
Scenario: Вход 5-го участника отклоняется
  Given в комнате уже 4 участника
  When пятый пользователь пытается войти
  Then система отклоняет подключение
  And показывает сообщение "Комната заполнена" с кнопкой повторить вход

Scenario: Одновременная гонка за последний слот
  Given в комнате 3 участника
  When двое пользователей пытаются войти одновременно
  Then серверная проверка лимита выполняется атомарно
  And в комнату попадает только один из них, второй получает "Комната заполнена"
```

### US-6 — Видео и аудио в реальном времени
**As a** участник, **I want to** видеть и слышать остальных участников, **so that** мы могли общаться «лицом к лицу».

```gherkin
Scenario: Потоки участников отображаются в сетке
  Given в комнате двое или более участников
  When звонок активен
  Then видеопотоки отображаются в адаптивной сетке (например, 2×2)
  And собственное видео показывается отдельно (self-view)
  And имя каждого участника выводится поверх его плитки

Scenario: Состояние медиа по умолчанию
  Given пользователь входит в комнату и дал доступ к устройствам
  Then его микрофон и камера включены по умолчанию

Scenario: Вход без физических устройств
  Given у пользователя физически нет камеры и/или микрофона
  When он входит в комнату
  Then он всё равно присоединяется
  And отсутствующие устройства у него выключены

Scenario: Задержка на локальной сети
  Given все участники в одной локальной сети
  When идёт аудио-видеообмен
  Then задержка медиа не превышает 500 мс
```

### US-7 — Управление микрофоном и камерой
**As a** участник, **I want to** включать/выключать свой микрофон и камеру, **so that** я контролировал, что слышат и видят остальные.

```gherkin
Scenario: Выключение микрофона
  Given микрофон участника включён
  When участник выключает микрофон
  Then его аудио перестаёт передаваться остальным
  And на его плитке отображается иконка перечёркнутого микрофона

Scenario: Выключение камеры освобождает устройство
  Given камера участника включена (индикатор-лампочка камеры горит)
  When участник выключает камеру
  Then видеодорожка освобождается, камера физически перестаёт использоваться, лампочка гаснет
  And на месте видео показывается заглушка с силуэтом человека и именем участника

Scenario: Потеря устройства во время звонка
  Given во время звонка выбранное устройство стало недоступно (отключили/занял другой апп)
  When устройство теряется
  Then соответствующий поток прекращается
  And для восстановления пользователь выбирает устройство в настройках браузера/ОС
```

### US-8 — Текстовый чат
**As a** участник, **I want to** отправлять и читать сообщения в общем чате, **so that** я мог делиться ссылками и информацией прямо во время звонка.

```gherkin
Scenario: Отправка сообщения
  Given участник находится в комнате
  When он вводит непустой текст и отправляет сообщение
  Then сообщение появляется в общем чате у всех участников в реальном времени
  And содержит имя отправителя и время в формате HH:MM по локальному времени клиента

Scenario: Пустое сообщение
  Given поле ввода пустое или содержит только пробелы
  When участник пытается отправить
  Then отправка не происходит

Scenario: Сообщение со вредоносным содержимым
  Given участник отправляет текст с HTML/JS-разметкой
  When сообщение отображается у других
  Then оно экранируется и не исполняется (защита от XSS)

Scenario: Автопрокрутка к новому сообщению
  Given в чате есть история сообщений
  When приходит новое сообщение
  Then чат автоматически прокручивается к последнему сообщению

Scenario: Поздний вход — видна предыдущая переписка
  Given в комнате уже идёт переписка
  When новый участник входит в эту комнату
  Then он видит сообщения, отправленные до его входа (за время жизни комнаты)
```

### US-9 — Список участников и системные события
**As a** участник, **I want to** видеть, кто сейчас в комнате и когда кто-то входит или выходит, **so that** я понимал состав звонка в любой момент.

```gherkin
Scenario: Актуальный список участников
  Given участник находится в комнате
  When состав комнаты меняется
  Then список участников обновляется в реальном времени

Scenario: Системные сообщения о входе и выходе
  Given участник находится в комнате
  When другой участник входит или покидает комнату
  Then в чат добавляется системное сообщение "участник присоединился / покинул комнату"
```

### US-10 — Выход и жизненный цикл комнаты
**As a** участник, **I want to** выйти из комнаты, **so that** я мог завершить звонок, не оставляя «зависшую» плитку для остальных.

```gherkin
Scenario: Осознанный выход
  Given участник находится в комнате
  When он нажимает кнопку "Выйти"
  Then он покидает комнату
  And его плитка исчезает у остальных
  And остальные получают системное сообщение о его выходе

Scenario: Закрытие вкладки и перезагрузка
  Given участник находится в комнате
  When он закрывает вкладку
  Then это приравнивается к выходу
  When он перезагружает страницу
  Then это считается новым входом (с повторным вводом имени)

Scenario: Удаление комнаты после выхода последнего участника
  Given в комнате остаётся один участник
  When он выходит или теряет соединение
  Then комната, её идентификатор и история чата полностью удаляются
  And повторный вход по тому же идентификатору создаёт новую пустую комнату
```

### US-11 — Обработка обрыва соединения
**As a** оставшийся участник, **I want to** чтобы при обрыве соединения у кого-то звонок продолжался, **so that** сбой у одного человека не ломал звонок для всех.

```gherkin
Scenario: Обрыв соединения участника
  Given в комнате несколько участников
  When у одного из них пропадает соединение
  Then его слот освобождается, его плитка исчезает
  And остальные продолжают звонок без прерывания
  And в чат добавляется системное сообщение, что участник вышел
  # Формулировку "соединение потеряно" не используем — мы не можем достоверно отличить обрыв от выхода

Scenario: Возврат после обрыва — только вручную
  Given участник выбыл из-за обрыва
  When он хочет вернуться
  Then он заново открывает ссылку и входит как новый участник
  And автоматическое переподключение не выполняется

Scenario: Несколько вкладок одного пользователя
  Given один пользователь открыл комнату в двух вкладках
  Then каждая вкладка считается отдельным участником и занимает слот
```

### US-12 — Отказ в доступе к устройствам
**As a** участник, **I want to** получать понятную обратную связь при отказе в доступе к камере/микрофону, **so that** я понимал, почему меня не видно/не слышно.

```gherkin
Scenario: Браузер не дал доступ к камере/микрофону
  Given пользователь входит в комнату
  When он отклоняет запрос на доступ к камере или микрофону
  Then показывается понятное сообщение об отсутствии доступа
  And пользователь остаётся в комнате с выключенными устройствами, не "вылетая" из приложения

Scenario: Как участника без видео видят остальные
  Given у участника нет видео (камера выключена, отсутствует или нет доступа)
  Then у остальных на его плитке отображается силуэт человека и его имя
  And если выключено аудио — на плитке отображается иконка перечёркнутого микрофона
```

### US-13 — Совместимость окружения и инфраструктурные ошибки
**As a** пользователь, **I want to** понимать, если что-то не так с браузером или сервером, **so that** я не упирался в «белый экран».

```gherkin
Scenario: Сервер недоступен
  Given приложение не может подключиться к серверу (сигнальный сервер недоступен)
  Then показывается понятное сообщение об ошибке сервера

Scenario: Браузер не поддерживает WebRTC
  Given пользователь открыл приложение в браузере без поддержки WebRTC или слишком старом
  Then показывается сообщение, что WebRTC не поддерживается

Scenario: Политика автозапуска (autoplay)
  Given браузер блокирует автозапуск аудио без действия пользователя
  When участник входит в комнату
  Then приложение запрашивает жест пользователя (например, действие входа / "включить звук")
  And после этого удалённое аудио воспроизводится
```

## 4. Functional Requirements

> Нумерованный список. В скобках — ID требования из тест-задания (трассируемость), где применимо. Приоритеты: **Must** — обязательно, **Should** — желательно.

**Вход, комнаты и жизненный цикл**
1. **(F-01, Must)** Система должна запрашивать отображаемое имя до входа в комнату и показывать его другим участникам.
2. **(F-02, Must)** При нажатии «Создать комнату» система должна генерировать уникальный идентификатор комнаты и перенаправлять пользователя на URL с этим идентификатором; этот URL служит ссылкой-приглашением.
3. **(F-03, Must)** Система должна позволять скопировать ссылку-приглашение (URL текущей комнаты).
4. **(F-04, Must)** При открытии URL с идентификатором комнаты система должна считывать идентификатор из URL и подключать пользователя к этой комнате после ввода имени.
5. **(Must)** Если комнаты с указанным идентификатором не существует, система должна моментально создать новую комнату с этим идентификатором и сделать пользователя её первым участником. Отдельного состояния «комната не найдена» не существует.
6. **(Must)** Вход в существующую комнату по угаданному/чужому идентификатору разрешён и считается штатным поведением — доступ к комнате не ограничивается (авторизации нет).
7. **(F-05, Must)** Система должна ограничивать комнату 4 участниками одновременно; проверка «не более 4» должна выполняться атомарно на сервере, чтобы одновременные подключения не превысили лимит.
8. **(Must)** 5-му подключению система должна отказывать с сообщением «Комната заполнена» и кнопкой повторить вход.
9. **(Must)** Комната должна храниться, пока в ней есть хотя бы один участник; при выходе/обрыве последнего участника комната, её идентификатор и история чата полностью удаляются. Повторный вход по тому же идентификатору создаёт новую пустую комнату.

**Видео и аудио**
10. **(F-06, Must)** Система должна передавать видео и аудио каждого участника через WebRTC (камера/микрофон).
11. **(F-07, Must)** Система должна отображать потоки в адаптивной сетке (1–4 плитки, например 2×2), показывая собственное видео отдельно (self-view).
12. **(F-08, Must)** Система должна выводить имя участника оверлеем поверх его плитки.
13. **(Must)** При входе в комнату микрофон и камера должны быть включены по умолчанию.
14. **(Must)** Если камера/микрофон физически отсутствуют, пользователь всё равно входит в комнату, но соответствующие устройства у него выключены.

**Микрофон, камера и индикация состояния**
15. **(F-09, Must)** Система должна позволять включать/выключать микрофон и показывать остальным индикатор его статуса.
16. **(Must)** При выключенном микрофоне на плитке участника должна отображаться иконка перечёркнутого микрофона.
17. **(F-10, Should)** Система должна позволять включать/выключать камеру.
18. **(Must)** При выключенной, отсутствующей или недоступной камере на месте видео должна отображаться заглушка с силуэтом человека и именем участника.
19. **(Must)** Выключение камеры должно освобождать видеодорожку (камера физически перестаёт использоваться, индикатор-«лампочка» камеры гаснет); включение — снова запускать её.
20. **(Must)** При потере выбранного устройства во время звонка соответствующий поток прекращается; восстановление выполняется выбором устройства в настройках браузера/ОС.

**Текстовый чат**
21. **(F-12, Must)** Система должна позволять отправлять сообщения в общий чат, видимые всем участникам в реальном времени.
22. **(F-13, Must)** У каждого сообщения система должна показывать имя отправителя и время в формате HH:MM по локальному времени клиента.
23. **(F-14, Must)** Система должна отображать историю сообщений текущей сессии и авто-прокручивать чат вниз при новом сообщении. Участник, вошедший позже, должен видеть сообщения, отправленные в комнату до его входа (за время жизни комнаты).
24. **(Must)** Система должна запрещать отправку пустого сообщения или сообщения из одних пробелов.
25. **(F-15, Should)** Система должна добавлять в чат системные сообщения о входе и выходе участников.

**Сессия и участники**
26. **(F-16, Must)** Система должна отображать актуальный список участников и обновлять его в реальном времени.
27. **(F-17, Must)** Система должна позволять выйти из комнаты: плитка исчезает у остальных, остальные получают системное сообщение о выходе.
28. **(Must)** Закрытие вкладки приравнивается к выходу; перезагрузка страницы считается новым входом (с повторным вводом имени).
29. **(Must)** Один пользователь может находиться в комнате одновременно в нескольких вкладках; каждая вкладка — отдельный участник и занимает слот.
30. **(Must)** Каждому участнику присваивается уникальный внутренний идентификатор (генерируется автоматически), не пересекающийся с другими и не отображаемый в интерфейсе; одинаковые отображаемые имена в одной комнате допускаются.
31. **(F-18, Must)** При обрыве соединения участник выбывает из звонка и чата, его слот освобождается, остальные продолжают звонок; в чат добавляется системное сообщение, что участник вышел (формулировку «соединение потеряно» не используем). Автопереподключение не выполняется — возврат только повторным входом.
32. **(Must)** У создателя комнаты нет никаких особых прав; все участники равны.

**Окружение, совместимость и инфраструктурные ошибки**
33. **(Must)** При отказе браузера в доступе к камере/микрофону система должна показывать понятное сообщение; пользователь остаётся в комнате с выключенными устройствами, не «вылетая».
34. **(Should)** Система должна корректно реагировать на недоступность STUN, не приводя приложение в нерабочее состояние.
35. **(Must)** Если не удаётся подключиться к серверу (сигнальный сервер недоступен), система должна показывать понятное сообщение об ошибке сервера.
36. **(Must)** Если браузер не поддерживает WebRTC или слишком старый, система должна показывать сообщение, что WebRTC не поддерживается.
37. **(Must)** Система должна учитывать политику автозапуска (autoplay) браузеров: поскольку воспроизведение удалённого аудио/видео может требовать жеста пользователя, приложение обеспечивает такой жест (например, действие входа / «включить звук»), после которого удалённое аудио воспроизводится.

**Валидация и безопасность**
38. **(Must)** Система должна ограничивать имя участника 30 символами и запрещать спецсимволы (с экранированием), чтобы исключить инъекции.
39. **(Must)** Система должна фильтровать/экранировать имена и текст сообщений для защиты от XSS при отображении в оверлеях и чате.
40. **(Should)** Лимит длины сообщения и/или защита от флуда — на усмотрение разработчика, с разумными значениями по умолчанию (как минимум — запрет пустых/пробельных сообщений, см. п. 24).

## 5. Non-Goals (Out of Scope)

Следующее **намеренно не входит** в объём данной версии:

- **Аутентификация и авторизация.** Достаточно одного отображаемого имени; аккаунтов, паролей, ролей нет. Доступ к комнате по идентификатору не ограничивается.
- **Любое сохранение состояния на клиенте.** Имя и какая-либо информация между перезагрузками/перезапусками не запоминаются (без localStorage и подобного).
- **Постоянное серверное хранение сообщений.** История чата живёт только в памяти на время жизни комнаты; после её удаления сообщения исчезают.
- **Автоматическое переподключение (auto-reconnect).** При обрыве участник выбывает и возвращается только вручную.
- **Индикатор активного говорящего.**
- **Привилегии модератора / создателя** (выключить чужой микрофон, удалить участника). Все участники равны.
- **Встроенный выбор устройств** (камера/микрофон) не обязателен; смена устройства — через настройки браузера/ОС.
- **TURN-сервер.** Для локальной и офисной среды достаточно публичных STUN; недостижимость отдельной пары из-за строгого NAT — допустима (см. раздел 7).
- **Нормирование качества/битрейта.** Потолок битрейта не задаётся — на усмотрение разработчика.
- **Звуки** входа, выхода и нового сообщения.
- **Мультиязычность (i18n).** Интерфейс только на русском.
- **Мобильная адаптация.** Целевые экраны — десктоп от 1024px.
- **Более 4 участников**, комнаты-«вебинары», лобби/«waiting room», breakout-rooms.
- **Демонстрация экрана, запись звонка, виртуальные фоны, реакции/эмодзи.**
- **Расширенный чат:** файлы/картинки, редактирование и удаление сообщений, личные сообщения, реакции.
- **Сквозное (E2E) шифрование** сверх штатного DTLS-SRTP, которое WebRTC обеспечивает по умолчанию.

## 6. Design Considerations

**Язык интерфейса — русский** (один язык, без i18n). **Референс по UX и фичам — рабочее демо `https://chat.forasoft.com`** («Video Chat Starter Kit»): мгновенный групповой звонок в браузере без регистрации и установки.

**Экраны.**

- **Стартовый экран:** поле ввода имени (до 30 символов, без спецсимволов); кнопка «Создать комнату». Открытие ссылки-приглашения ведёт на экран комнаты после запроса имени.
- **Экран комнаты:** видеосетка (до 4 плиток); панель управления (микрофон, камера, выход); панель/оверлей с текстовым чатом и списком участников.
- **Состояния и заглушки:** при отсутствии видео — силуэт человека с именем; при выключенном микрофоне — иконка перечёркнутого микрофона на плитке; при заполненной комнате — сообщение «Комната заполнена» с кнопкой повторить вход; понятные сообщения при отказе в доступе к устройствам, недоступности сервера и отсутствии поддержки WebRTC.

**Принципы оформления.**

- Адаптивная раскладка корректно работает на ширине от 1024px.
- Видеосетка перестраивается под число участников (1 / 2 / 3 / 4 плитки), например 2×2.
- Визуальный дизайн (CSS) оценивается по принципу «работает и понятно», а не по эстетике; приоритет — читаемость и очевидность элементов управления.

## 7. Technical Considerations

**Обязательный стек (тест-задание, раздел 6.1):** JavaScript (ES6+), Node.js (сервер), React (UI), Socket.io (сигналинг и текстовый чат), WebRTC (передача видео/аудио).

**Архитектура (зафиксированный выбор для этого PRD):**

- **Топология медиа — WebRTC mesh (peer-to-peer).** Каждый участник держит прямое P2P-соединение с каждым другим: при N участниках это N·(N−1)/2 соединений (для 4 участников — 6 соединений на комнату, по 3 исходящих потока на клиента). Именно поэтому лимит участников зафиксирован на уровне **4**.
- **Сигнальный сервер — Socket.io.** Сервер на Node.js + Socket.io обменивается между клиентами SDP-офферами/ответами и ICE-кандидатами, обслуживает текстовый чат и системные события, а также атомарно контролирует лимит «не более 4» при подключении.
- **ICE / NAT-обход — публичные Google STUN** (`stun.l.google.com`). TURN не используется; недостижимость отдельной пары из-за строгого NAT допустима и обрабатывается на усмотрение разработчика.
- **Состояние комнат и участников — в памяти сервера.** Без БД и без персистентного хранилища. На клиенте ничего не сохраняется между перезагрузками.

**Поведение сессии и данных:**

- **Жизненный цикл комнаты.** Комната создаётся при первом участнике (в т. ч. при входе по новому идентификатору из URL) и существует, пока в ней есть хотя бы один участник; при падении счётчика до нуля комната, её идентификатор и история чата удаляются.
- **Идентификаторы участников.** Внутренний ID генерируется автоматически (например, socket-id или UUID), уникален в пределах комнаты и не отображается в UI; отображаемые имена могут повторяться.
- **Тумблер камеры.** Выключение камеры должно останавливать видеодорожку (а не только отключать её), чтобы аппаратный индикатор камеры гас; включение — пересоздавать дорожку.
- **Обрыв соединения.** Участник удаляется из комнаты (звонок + чат); автопереподключение не реализуется.
- **Несколько вкладок.** Разрешены; каждая вкладка — независимый участник.

**Окружение и совместимость:**

- **HTTPS обязателен.** `getUserMedia` и WebRTC работают только в защищённом контексте (HTTPS или localhost).
- **Поддержка браузеров:** Chrome 100+, Firefox 100+, Edge 100+; при отсутствии поддержки WebRTC — явное сообщение.
- **Политика autoplay.** Удалённое аудио может не воспроизводиться без жеста пользователя — приложение обеспечивает такой жест (действие входа / «включить звук»).
- **Задержка** видео/аудио ≤ 500 мс в локальной сети. Потолок битрейта не нормируется.

**Безопасность:**

- Имена (≤30 символов, без спецсимволов) и текст сообщений экранируются для защиты от XSS.
- Доступ к комнате по идентификатору намеренно не ограничивается (нет авторизации) — это осознанный компромисс для сценария «зашёл по ссылке».

**⚠️ Важно — расхождение с референс-демо.** Демо `chat.forasoft.com` фактически построено на **LiveKit (SFU) + Next.js + Express + Socket.IO**, то есть использует медиасервер (SFU), который сам берёт на себя сигналинг, маршрутизацию медиа и ICE. В данном PRD **намеренно зафиксирован стек тест-задания**: «чистый» WebRTC в топологии mesh, Socket.io в роли сигнального сервера, публичные Google STUN. Демо используется как референс по **фичам и UX**, но **не по архитектуре медиаслоя**.
